# Codexで「ほとんどなんでも作れる」感覚を持つ授業案

## 授業の位置づけ

この回の一番の目標は、学生がCodexを使えば「ほとんどなんでも作れるかもしれない」と実感することです。

ここで言う「作れる」は、最初から完成度の高いものを一人で設計できるという意味ではありません。Codexに頼みながら、まず動くものを作り、触ってみて、「ここを変えたい」「ここが分かりにくい」「この機能を足したい」と言葉にし、少しずつ直していけるという意味です。

この回は、教員が前で講義をして全員が同じ速度で進む形式ではありません。学生は教材ページを見ながら各自で進め、分からないところがあったら教員やTAに質問します。

2週間後までの宿題として、実際に使える小さなサービスを1つ作ってきます。デプロイは必須にしません。自分のPC上で動き、使い方を説明できればよいことにします。

## 対象学生の前提

- 商学部3年生
- プログラミング経験はない、またはほとんどない
- ターミナル、ローカルサーバー、フォルダ構成などの言葉には馴染みがない
- AIチャットの利用経験はあるかもしれないが、AIにコードを書かせて動かす経験は少ない
- 「失敗したらどうしよう」「変なものを作ってしまったらどうしよう」という不安が出やすい

この前提に合わせて、教材では次を大事にします。

1. コードの説明から入らない
2. まず「何が作れそうか」を広く見る
3. 作りたいものを、学生自身の生活・関心・商学部らしい問題意識に引き寄せる
4. 画面で動くものを小さく作る
5. 作ったものを触って、試しにいじる
6. うまくいかないときに、何をCodexへ伝えればよいかを示す
7. 「完成」より「直し続ける」を重視する

## 進め方の前提

この回では、進捗にかなり差が出る前提で教材を設計します。

- 進みがゆっくりな学生は、Codexの準備と、事例を1つ動かすところまで行ければよい
- 標準的な学生は、事例を1つ作り、少なくとも1回いじって直す
- 進みが早い学生は、事例を2つ以上作るか、元プロンプトを改変して自分のサービス案に近づける

そのため、教材は「全員が同じページを同じ時間に読む」構成にはしません。最初の導線は共通にしつつ、途中からは事例集を選んで進められる形にします。

各事例には、次の3つを置きます。

1. そのまま貼れる元プロンプト
2. 自分向けに変えるための改変ポイント
3. 作ったあとに試す追加依頼

## 今日のゴール

授業終了時点で、学生が次の状態になっていることを目指します。

1. Codexを準備し、作業フォルダで使い始められる
2. ブラウザで動く小さなWebサービスを1つ作り、開いて触れる
3. 元プロンプトを改変し、できたものを1回以上直せる

## 今日の目標

今日の教材ページでは、次の3つを明確に示します。

1. Codexを使えば、Webサービスに限らず、アプリやカメラを使う道具のようなものまで作り始められると知る
2. 授業時間内では、まずWebブラウザで動く小さなWebサービスを1つ作る
3. `AGENTS.md` を作業フォルダごとに置き、自分で調べながら進める準備をする

### 2. 何ができるのかを知る

URL案: `20260528/can-make/`

このページでは、Codexで作れるものの幅を最初に見せます。

最初に伝えること:

- Codexで作れるものは、Webサイトだけではない
- やろうと思えば、Androidアプリ、iPhoneアプリ、PCのカメラを使う道具、外部機器と組み合わせた試作も作り始められる
- ただし、スマホアプリやカメラを使うものは、環境設定や実機確認など、自分で手を動かす工程が増える
- 今日の授業時間では、まず一番始めやすく、ブラウザですぐ確認できるWebサービスから始める

学生に見せる例:

- Webサービス: 自己紹介、商品紹介、クイズ、記録、比較ツール
- チャット風アプリ: 質問に答える、練習相手になる
- Androidアプリ: スマートフォンで動くアプリ
- iPhoneアプリ: iPhoneで動くアプリ
- カメラを使う道具: PCやスマートフォンのカメラを使って撮影、記録、判定をする
- 外部機器を使う試作: センサー、ボタン、カメラなどを組み合わせる

伝えること:

- どれも最初は小さく作る
- 最初から本格的なアプリにしなくてよい
- 「誰が、どんな場面で、何に困っているか」を書くと作りやすい
- 作ったあとに触って直すことで、自分のサービスに近づいていく
- 早く進んだ人は、別の事例を作るか、元プロンプトを改変する

学生向けの書き方例:

> Codexで作れるものは、Webサイトだけではありません。スマホアプリやカメラを使う道具も作り始められます。ただし今日は、全員がまず1つ動かせるように、ブラウザで確認しやすいWebサービスから始めます。

### 4. 共通の制作フローを確認する

URL案: `20260528/workflow/`

このページでは、事例を使う場合にも、自分で作りたいものを書く場合にも共通する進め方を説明します。

ページ冒頭に長い手順リストは置きません。対象学生には急に見えてしまうため、次のように番号付きの見出しで、上から読めばそのまま進められる形にします。

1. 作業フォルダを用意する
2. `AGENTS.md` をそのフォルダに入れる
3. Codexでそのフォルダを選ぶ
4. チャットに作りたいものを書いて送る
5. 実際に動くか試す
6. 開いて触る
7. 気になるところを伝えて直す
8. この流れをくり返す

学生向けの書き方例:

> 事例ページのプロンプトを使ってもよいですし、自分で作りたいものをそのまま書いても構いません。大事なのは、最初から完璧に頼むことではなく、作って、動かして、触って、直すことです。

注意:

- `AGENTS.md` は、ダウンロードしただけでは効かない。作業フォルダの中に入れる
- 別のサービスを作るために新しいフォルダを作ったら、そのフォルダにも `AGENTS.md` を入れる
- Codexのチャット画面で、必ずいま作るものの作業フォルダを選ぶ
- チャットには、自分のレベルではなく、課題、解決策、目的、具体的な動きを書く
- まだ見えていない部分は空欄でもよい。作ってから直せることを伝える
- 動かし方が分からなかったら、すぐCodexに聞く
- `localhost` や「サーバーを起動する」のような言葉は、このページ内で短く説明する
- 開けないときにエラー文を貼って聞くプロンプトも、このページ内に置く
- 直すときは大きく作り替えすぎず、まず気になるところを1つだけ伝える
- 「気になるところ」「残したいところ」を書ける修正依頼の型も、このページ内に置く

### 5. 自分用の `AGENTS.md` を作る

URL案: `20260528/agents-md/`

このページでは、Codexに毎回読んでほしい自分用の指示を作ります。

説明方針:

- `AGENTS.md` は、Codexへの「このフォルダではこう進めてほしい」というメモ
- Codexは作業を始めるときに、そのフォルダに置かれた `AGENTS.md` を読む
- 自分の前提、説明してほしい細かさ、進め方の好みを書いておくと、毎回のチャットで同じことを説明しなくてよくなる
- `AGENTS.md` は作業フォルダごとに置く。別のサービスを作る新しいフォルダを作ったら、そのフォルダにも入れる
- ダウンロードしたら、いま作るサービスの作業フォルダに入れる
- 最初から完璧に書かなくてよい
- 自分の進み具合に合わせて更新してよい

入れる内容:

- 自分の経験レベル
- 説明の細かさ
- コードを勝手に変えてよい範囲
- 作ったあとに必ず教えてほしいこと
- エラーが出たときの進め方
- デザインの好み

教材ページには、いきなり編集できる `AGENTS.md` のドラフトをテキスト欄で出します。学生はその本文を直接書き換え、下にある「AGENTS.mdをダウンロード」「内容をコピー」ボタンを使います。

編集途中の本文はブラウザの `localStorage` に保存し、ページをリロードしたり、別ページに移動して戻ったりしても消えないようにします。

### 8. 事例URLから小さく作ってみる

URL案: `20260528/first-build/`

このページでは、事例一覧から個別URLを1つ選び、短時間で作ります。

候補は並列に扱います。教員が全員に同じ題材を選ばせたい場合は「自分を表すWebサイト」を使えます。ただし、教材上の位置づけは「Codexで作れるものを体験するための最初の練習」です。進みが早い学生は、同じ事例を改変するか、別の事例URLをもう1つ開いて作ります。

最初に出す選択肢:

1. 自分を表すWebサイト
2. インスタのストーリー風アプリ
3. BeReal風: 今日の一枚
4. MBTI/相性診断風ジェネレーター

最初は、次の条件を満たせば成功とします。

- 作業フォルダに `AGENTS.md` が入っている
- 事例ページのプロンプトを、自分の作りたいものに合わせて少しでも書き換えようとしている
- 自分のPCで開ける
- 画面上で何か操作できる、または内容を読める
- 何をするものか一言で説明できる
- Codexに「どうやって動かすか」を聞けている

画像を使う場合、授業中の練習では仮の画像やプレースホルダーでも構いません。ただし、本番として提出するものでは、Computer Use + 編集で、自分で必要な画像を作るか、編集して差し替えることを明記します。Webから拾った画像を理由なくそのまま使う形にはしません。

各事例の具体的な元プロンプトは、個別事例ページに置きます。このページでは、どれを選ぶかと、共通の進め方だけを確認します。

URL案: `20260528/homework/`

宿題:

> 2週間後までに、自分または誰かが実際に使える小さなサービスを1つ作ってください。

サービスの条件:

- 自分のPCで動く
- 使い方を説明できる
- 誰のどんな場面で役に立つかを説明できる
- 最低3回は改善している
- 改善前と改善後の違いを説明できる
- 画像を使う場合は、Computer Use + 編集で自分で作る、または自分で編集したものに差し替えている

作るものの例:

- 推し活の熱量を管理するメーター
- 一瞬だけ集まって解散する共同体ルーム
- 自分好みに調整できるローカルAIメモ

サービス案そのものは、教材ページだけで決めさせず、人間がメンタリングしながら相談して決める想定です。

提出で見たいもの:

- サービス名
- 誰のためのものか
- 何ができるか
- 自分のPCでどう動かすか
- 3回以上直した内容

### 10. 提出前チェック

URL案: `20260528/submission/`

提出前に学生が確認するページです。今日の授業内では、最後に作ったもののスクリーンショットをDiscordに投稿します。

チェック項目:

- サービスを自分のPCで開ける
- 何をするサービスか一言で説明できる
- 使い方を3ステップ以内で説明できる
- 3回以上改善した記録がある
- 個人情報や公開してはいけない情報を入れていない
- Codexに聞いたことを、自分の言葉で少し説明できる

今日の最後にやること:

1. 作ったものをブラウザで開く
2. 画面のスクリーンショットを撮る
3. Discordの指定チャンネルに投稿する
4. できれば、何を作ったかを一言添える
